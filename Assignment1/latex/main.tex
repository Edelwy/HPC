Our scripts and modified source code can be found in our \href{https://github.com/Edelwy/HPC}{\textbf{GitHub} repository}. \href{https://en.wikipedia.org/wiki/Seam_carving}{\textbf{Seam carving}} resizes an image by repeatedly finding a low-energy vertical seam, removing one pixel per row, and shifting the remaining pixels. Each outer iteration applies three phases in order: we compute a the Sobel energy map, find the cheapest seam by dynamic programming and trace the cheapest seam from the top row down.

\section{Implementation}

The baseline parallel version uses \href{https://www.openmp.org/wp-content/uploads/OpenMP-4.0-C.pdf}{\textbf{OpenMP}} library and can be enabled via \texttt{USE\_OMP} macro. This includes parallization per pixel in the energy calculation, parallelization per row in seam identification and parallelization in pixel removal per seam partitions which are dependant on height. An \textbf{optimization} is supported via \texttt{USE\_DYNAMIC\_THREADS} macro, which keeps the same pragmas but sets \textbf{different number of threads} at the boundaries between phases so each stage can use a different size. The heuristic dynamically selects the number of threads based on width and height. For the energy computation, the thread count scales with the total workload divided by a tuning constant \texttt{THREADS\_ENERGY\_CONSTANT (C1)}. The thread count for the dynamic programming phase is derived from the width and \texttt{THREADS\_DP\_CONSTANT (C2)} and capped at half of the available threads to reduce synchronization overhead. Similarly, the seam removal phase is memory-bound and parallelized across image rows, so its thread count is based on the image height and \texttt{THREADS\_REMOVAL\_CONSTANT (C3)} but not capped at half like before. The constants were set to different valaues and tested to find the right balance as can be seen in results. We name thread counts implied by these constants \texttt{T1}, \texttt{T2} and \texttt{T3} respectively.

We also implemented \textbf{greedy seam removal} which removes multiple seams without calculating the energy for each removal and compared the results by looking at speed up and picture quality. In order to achieve further paralelization we divided the picture into blocks where width is divided by the number of threads in use. The path finding is limited to the block so that these can be done in parallel independently. This is supported via \texttt{USE\_GREEDY\_REMOVAL} macro.

\section{Dynamic Thread Optimization}

We ran the whole pipeline using four times for 16 and once 32 threads, setting each of the constants to different values. Afterwards we tweaked each individual value again. We tried to figure out whether a low thread or a high thread count was prefered by each step. The \textbf{best values for the constants and average time for optimized and non-optimized algorithm} in seconds for 16 threads can be seen below. Full results and analysis can be found in our \href{https://github.com/Edelwy/HPC/blob/main/Assignment1/latex/analiza.ipynb}{jupyter notebook}.

Since pixel count is quite high for all images, the expected result is to use all \textbf{16 threads} for energy calculation. For the seam identification almost always \textbf{less threads} were prefered, for the smallest images always choosing one thread as the optimal solution. So due to sychronization overhead a sequential approach to seam finding for small images turned out to be better. It is interasting to see that for all images \textbf{high thread} count was prefered when shifting pixles, meaning most threads were utilized, therefore memory constraints did not impose much overhead.
